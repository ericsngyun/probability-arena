# SAFETY_BOUNDARIES — hard limits and their gates

These are project-level boundaries, not suggestions. Any task that appears to
require crossing one must stop and report back instead of building.

## Forbidden today (no implementation surface may exist)

| Boundary | Status | Gate: what must be explicitly accepted first |
|---|---|---|
| **EV calculation** | ❌ none exists | The MVP-005A gate crossed (paired n=36, both deltas negative — 2026-07-04); the accepted design was implemented as the **edge precheck**: probability-gap measurement (forecast − midpoint) with validity checks, behind `ENABLE_EDGE_PRECHECK=false`. It has no dollar-EV, side, size, or action fields by construction, and `paper_candidate_later` is a review label with zero behavior. **Dollar EV remains forbidden with no unlocking milestone defined** |
| **Trade recommendations** | ❌ none exists | Post-MVP-005B review; never from a forecaster directly — forecasts are reasoning artifacts |
| **Paper trading / simulation** | ❌ none exists | MVP-005B, gated on MVP-005A acceptance. EDGE-ANALYSIS-001 (`edge-cohort-report`) only *measures* per-cohort gap follow-through and *reports* whether the MVP-005B gate is met — it is analysis, unlocks nothing, and advancing still requires explicit human acceptance |
| **Portfolio sizing** | ❌ none exists | Post-paper-trading milestone with explicit human acceptance |
| **Order placement** | ❌ none exists | A dedicated, explicitly-accepted live-trading milestone (not currently on the roadmap) |
| **Wallet / private-key handling** | ❌ none exists (ADR-002) | A dedicated custody design + security review milestone; keys would never live in this repo/DB regardless |
| **Live trading / execution** | ❌ none exists | Same as order placement; also requires operational controls (limits, kill switches) designed first |
| **Autonomous trading** | ❌ none exists | Not planned; would require all of the above plus standing human-in-the-loop controls |
| **Crypto wallets** | ❌ none exists | CRYPTO-001 shipped read-only scouting only; wallet milestones explicitly deferred |
| **Swaps / transaction construction / signing (Jupiter or any DEX)** | ❌ none exists | WALLET-001 (policy-controlled transaction *proposal* gateway only — no signing/keys), itself gated on CRYPTO-002 (risk engine) + CRYPTO-003 (paper simulator) acceptance; much later |

## What "no implementation surface" means

- No functions, fields, tables, endpoints, or CLI commands for these capabilities — including "disabled" or "placeholder" versions.
- `paper_candidate_pending` is a human review **label** on signals with zero attached behavior; keep it that way.
- The safety grep in `AGENTS.md` / `docs/TESTING_POLICY.md` must come back clean before any milestone is declared done.

## Always true, phase-independent

- All external interaction is read-only (GETs); Kalshi credentials are not required and not stored; the optional WS client sends channel subscriptions only.
- Crypto Arena (CRYPTO-001) is read-only surveillance: public DEX data in, auditable rows out. Its signals are telemetry — no wallet code, no private keys, no swaps, no Jupiter/transaction construction, no signing, no execution, and no EV/paper-trading semantics may attach to them before the gated milestones above.
- The crypto risk engine (CRYPTO-002) produces **risk intelligence, not trade advice**: a composite risk score/level is an avoid/flag verdict for human review. "Severe" means avoid/flag — never short/sell/buy. Provider API keys are optional, header-only, and never printed or logged. The MEME-RISK-003 holder/sniper/insider/bundler/creator coverage extension (Birdeye provider + coverage reports) is likewise read-only risk intelligence: it does not trade, paper trade, compute EV, recommend, size, or place orders, and uses no wallets/keys/swaps/signing/execution. Provider absence is reported explicitly (a coverage gap is stated, never silently treated as "clean"); keys are reported present/absent only, never their values.
- The memecoin label follow-through analysis (MEME-SHADOW-001) is **read-only calibration MEASUREMENT only**: it reconstructs the MEME-MAS `review_priority` at each historical attention snapshot (reusing the deterministic MEME-MAS agents with the risk assessment as-of that moment) and measures how the SAME token moved afterwards (price/liquidity/volume change at 5m/15m/1h/6h/24h, survival, rug/liquidity-removed incidence, attention persistence, risk-level transition) from the token's own later snapshots. `price_change` is measured MARKET MOVEMENT of the token, exactly like the edge follow-through analysis — **not PnL, not a fill, not EV, not paper trading, not a trade recommendation, not position sizing, not a profit expectation**. Its only output is whether `review_priority` (and sub-scores / risk reasons / concentration) SEPARATE later behavior, plus a conservative calibration recommendation (`too_thin_to_calibrate` / `labels_separate_outcomes` / `no_material_separation_recalibrate` / `review_priority_inverted_recheck`). It changes no label and authorizes nothing. Derived read-only on demand; nothing persisted, no external call, no provider-budget impact; MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket behavior unchanged.
- The memecoin multi-agent diagnostic layer (MEME-MAS-001) is **read-only DIAGNOSTIC intelligence only**: five deterministic "agents" (Coin Structure, Catalyst Velocity, Timing, Risk Auditor, Composite Review — pure functions, **no LLM, no external calls, no new providers**) recompute sub-scores on demand from already-persisted rows (`meme_attention_snapshots`, `crypto_token_risk_assessments`, `meme_catalyst_events`). Its output `review_priority` (`low`/`monitor`/`elevated_review`/`high_review`/`reject_risk`) triages HOW MUCH HUMAN REVIEW a token warrants — it is **not a trade recommendation, not EV, not a position size, not a profit expectation, not a buy/sell/trade/bet signal, and not an instruction**. The MEME-MAS-002 recalibration (profile `v2`, with `v1` preserved only for the `meme-mas-calibration-report` before/after comparison) makes `high_review` risk-aware and gated (requires clean structure, non-missing provider coverage, and no concentration flags) and adds `momentum_quality`/`structure_quality`/`coverage_quality` diagnostic outputs — it is **label calibration only**, still read-only, still no EV/trade/paper/sizing/orders/wallets/keys/signing/swaps/execution, and `reject_risk` remains a hard avoid/flag gate. `reject_risk` is an avoid/flag verdict for review, never a trade direction. It computes no EV, does no paper trading, sizes no positions, places no orders, and uses no wallets/private keys/swaps/signing/execution. It adds no external request and no provider-budget impact (derived from existing rows); it changes no MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket behavior; persists nothing (no new table).
- The SolanaTracker request budget (PROVIDER-BUDGET-001) is **provider cost/usage OBSERVABILITY only**: request counts are derived read-only from persisted risk assessments, and `crypto-provider-budget-report` reports usage against the plan (SolanaTracker Advanced ≈ $58–59/month USD, 200,000 requests/month — an accounting/ops metadata note, NOT a PnL/EV/profit feature). The budget guardrail can only **skip** optional SolanaTracker lookups when over the per-run cap or daily STOP threshold — a skipped token falls back to GoPlus + heuristics (a supported mode). It never adds calls, never changes GoPlus/Birdeye behavior, and attaches no EV, trade recommendation, position sizing, order, wallet/private key, signing, swap, execution, or autonomy. It cannot silently exceed budget (skips are logged; the report shows the WARN/STOP state explicitly).
- No secrets in code, logs, or committed files; `.env` is gitignored.
- Forecast confidence is capped centrally by evidence depth; forecasters cannot self-declare certainty. Evidence-aware forecasts (baseball, soccer, and the TENNIS-001 tennis canary) are measurement inputs only — never advice, never sized, never actionable. TENNIS-001 is read-only evidence and forecasting: it does not trade, paper trade, compute EV, recommend bets, size positions, place orders, or use wallets/private keys/swaps/signing/execution; its `espn` provider is read-only and its payload mapping is pending validation (honest template fallback until then).
- Every model-assisted path has a deterministic fallback and honest degradation (template content stays labeled template).
- The scheduled meme/news lane (MEME-NEWS-002) is **read-only scheduled discovery only**: a bounded systemd-timer runner around the MEME-NEWS-001 scout, plus a windowed report and local derived alerts. `attention_score` is not EV, not a recommendation, and not an instruction; alerts are informational local report rows (no push notifications, no trade triggers, no sizing). A severe/high-risk alert is an avoid/flag verdict for review — never a trade direction. `ENABLE_MEME_NEWS_SCOUT` gates only the scheduled path (manual runs always allowed); the lane is a separate process that cannot affect MarketOps/EDGE-AUTO. No wallets/private keys, swaps, signing, orders, execution, position sizing, paper trading, or live trading exist.
- The Polymarket market-data observer (POLY-001) is **read-only market-DATA observation only**: a bounded scan over the public, no-authentication Polymarket endpoints (the Gamma market catalog + the CLOB read-only order book), persisting market/orderbook/domain-inventory snapshots plus a windowed report and a per-domain inventory report. It exists to observe a SECOND prediction-market venue (Polymarket) alongside Kalshi. Prices and order books are informational quotes for human review — not EV, not a recommendation, not an instruction, not a trade trigger. It uses **no API keys, no wallets/private keys, no signing** and requires none; the authenticated CLOB trading endpoints (order create/cancel, signing) are deliberately **not implemented**. `ENABLE_POLYMARKET_SCOUT` gates only any future scheduled path (manual runs always allowed); the lane is a separate process that cannot affect MarketOps/EDGE-AUTO. Cross-venue semantic linking to Kalshi is a documented **POLY-002 placeholder only** — no `comparable_market_candidate` / `unresolved_semantic_match` / `cross_venue_observation` rows are produced, and **no arbitrage, EV, or trade-candidate labels exist or are implied**. No EV, arbitrage execution, trade recommendations, position sizing, order placement, wallets/private keys, signing, swaps, paper trading, live trading, or autonomous trading exist.
- The meme/news + domain-expansion scout (MEME-NEWS-001) is **read-only discovery, scouting, and inventory only**: it does not trade, paper trade, compute EV, recommend trades, size positions, or place orders, and uses no wallets/private keys/swaps/signing/execution. An `attention_score` is an interest/velocity signal for human review — never a buy/trade/EV/alpha score, and it triggers no behavior. Catalyst events are informational, never a trade trigger; non-dexscreener sources (rss/x/discord/telegram) are schema placeholders added only if explicitly configured (no authenticated scraping). The domain scout adds no forecaster and changes no promotion/edge/forecast logic — `canary_priority` ranks candidate domains for future human-planned canaries only.
- The frontier evaluation harness (EVAL-001) is evaluation only: gap follow-through is market-movement analysis, not PnL; no fills or positions are simulated; readiness labels (`not_ready` … `ready_for_paper_design`) gate further MEASUREMENT milestones and never authorize live capital — no live/autonomous readiness label exists by design.
