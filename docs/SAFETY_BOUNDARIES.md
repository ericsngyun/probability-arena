# SAFETY_BOUNDARIES — hard limits and their gates

These are project-level boundaries, not suggestions. Any task that appears to
require crossing one must stop and report back instead of building.

## Forbidden today (no implementation surface may exist)

| Boundary | Status | Gate: what must be explicitly accepted first |
|---|---|---|
| **EV calculation** | ❌ none exists | The MVP-005A gate crossed (paired n=36, both deltas negative — 2026-07-04); the accepted design was implemented as the **edge precheck**: probability-gap measurement (forecast − midpoint) with validity checks, behind `ENABLE_EDGE_PRECHECK=false`. It has no dollar-EV, side, size, or action fields by construction, and `paper_candidate_later` is a review label with zero behavior. **Dollar EV remains forbidden with no unlocking milestone defined** |
| **Trade recommendations** | ❌ none exists | Post-MVP-005B review; never from a forecaster directly — forecasts are reasoning artifacts |
| **Paper trading / simulation** | ❌ none exists | MVP-005B, gated on MVP-005A acceptance. EDGE-ANALYSIS-001 (`edge-cohort-report`) only *measures* per-cohort gap follow-through and *reports* whether the MVP-005B gate is met — it is analysis, unlocks nothing, and advancing still requires explicit human acceptance |
| **Portfolio sizing** | ❌ none exists | Post-paper-trading milestone with explicit human acceptance |
| **Order placement** | ❌ none exists | A dedicated, explicitly-accepted live-trading milestone (not currently on the roadmap) |
| **Wallet / private-key handling** | ❌ none exists (ADR-002) | A dedicated custody design + security review milestone; keys would never live in this repo/DB regardless |
| **Live trading / execution** | ❌ none exists | Same as order placement; also requires operational controls (limits, kill switches) designed first |
| **Autonomous trading** | ❌ none exists | Not planned; would require all of the above plus standing human-in-the-loop controls |
| **Crypto wallets** | ❌ none exists | CRYPTO-001 shipped read-only scouting only; wallet milestones explicitly deferred |
| **Swaps / transaction construction / signing (Jupiter or any DEX)** | ❌ none exists | WALLET-001 (policy-controlled transaction *proposal* gateway only — no signing/keys), itself gated on CRYPTO-002 (risk engine) + CRYPTO-003 (paper simulator) acceptance; much later |

## What "no implementation surface" means

- No functions, fields, tables, endpoints, or CLI commands for these capabilities — including "disabled" or "placeholder" versions.
- `paper_candidate_pending` is a human review **label** on signals with zero attached behavior; keep it that way.
- The safety grep in `AGENTS.md` / `docs/TESTING_POLICY.md` must come back clean before any milestone is declared done.

## Always true, phase-independent

- All external interaction is read-only (GETs); Kalshi credentials are not required and not stored; the optional WS client sends channel subscriptions only.
- Crypto Arena (CRYPTO-001) is read-only surveillance: public DEX data in, auditable rows out. Its signals are telemetry — no wallet code, no private keys, no swaps, no Jupiter/transaction construction, no signing, no execution, and no EV/paper-trading semantics may attach to them before the gated milestones above.
- The crypto risk engine (CRYPTO-002) produces **risk intelligence, not trade advice**: a composite risk score/level is an avoid/flag verdict for human review. "Severe" means avoid/flag — never short/sell/buy. Provider API keys are optional, header-only, and never printed or logged. The MEME-RISK-003 holder/sniper/insider/bundler/creator coverage extension (Birdeye provider + coverage reports) is likewise read-only risk intelligence: it does not trade, paper trade, compute EV, recommend, size, or place orders, and uses no wallets/keys/swaps/signing/execution. Provider absence is reported explicitly (a coverage gap is stated, never silently treated as "clean"); keys are reported present/absent only, never their values.
- The MEME-MAS-003 multi-objective calibration metrics extend MEME-SHADOW with five SEPARATE read-only measurement axes (`momentum_followthrough`, `survival_quality`, `risk_adjusted_movement`, `review_queue_efficiency`, `coverage_quality`) reported v1-vs-v2 by `meme-mas-objectives-report` — because a single survival yardstick misjudges volatile high-momentum review tiers. Each is a MEASUREMENT of the token's market movement / survival split by objective; `risk_adjusted_movement` is a measured median move discounted by observed survival — a **label-quality diagnostic, never a return, EV, PnL, or a fill**. It **changes no label** (MEME-MAS scoring untouched), authorizes nothing, and adds no EV/trade/paper/sizing/orders/wallets/keys/signing/swaps/execution; compute-on-demand, no external call, no budget impact.
- The memecoin label follow-through analysis (MEME-SHADOW-001) is **read-only calibration MEASUREMENT only**: it reconstructs the MEME-MAS `review_priority` at each historical attention snapshot (reusing the deterministic MEME-MAS agents with the risk assessment as-of that moment) and measures how the SAME token moved afterwards (price/liquidity/volume change at 5m/15m/1h/6h/24h, survival, rug/liquidity-removed incidence, attention persistence, risk-level transition) from the token's own later snapshots. `price_change` is measured MARKET MOVEMENT of the token, exactly like the edge follow-through analysis — **not PnL, not a fill, not EV, not paper trading, not a trade recommendation, not position sizing, not a profit expectation**. Its only output is whether `review_priority` (and sub-scores / risk reasons / concentration) SEPARATE later behavior, plus a conservative calibration recommendation (`too_thin_to_calibrate` / `labels_separate_outcomes` / `no_material_separation_recalibrate` / `review_priority_inverted_recheck`). It changes no label and authorizes nothing. Derived read-only on demand; nothing persisted, no external call, no provider-budget impact; MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket behavior unchanged.
- The memecoin multi-agent diagnostic layer (MEME-MAS-001) is **read-only DIAGNOSTIC intelligence only**: five deterministic "agents" (Coin Structure, Catalyst Velocity, Timing, Risk Auditor, Composite Review — pure functions, **no LLM, no external calls, no new providers**) recompute sub-scores on demand from already-persisted rows (`meme_attention_snapshots`, `crypto_token_risk_assessments`, `meme_catalyst_events`). Its output `review_priority` (`low`/`monitor`/`elevated_review`/`high_review`/`reject_risk`) triages HOW MUCH HUMAN REVIEW a token warrants — it is **not a trade recommendation, not EV, not a position size, not a profit expectation, not a buy/sell/trade/bet signal, and not an instruction**. The MEME-MAS-002 recalibration (profile `v2`, with `v1` preserved only for the `meme-mas-calibration-report` before/after comparison) makes `high_review` risk-aware and gated (requires clean structure, non-missing provider coverage, and no concentration flags) and adds `momentum_quality`/`structure_quality`/`coverage_quality` diagnostic outputs — it is **label calibration only**, still read-only, still no EV/trade/paper/sizing/orders/wallets/keys/signing/swaps/execution, and `reject_risk` remains a hard avoid/flag gate. `reject_risk` is an avoid/flag verdict for review, never a trade direction. It computes no EV, does no paper trading, sizes no positions, places no orders, and uses no wallets/private keys/swaps/signing/execution. It adds no external request and no provider-budget impact (derived from existing rows); it changes no MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket behavior; persists nothing (no new table).
- The SolanaTracker request budget (PROVIDER-BUDGET-001) is **provider cost/usage OBSERVABILITY only**: request counts are derived read-only from persisted risk assessments, and `crypto-provider-budget-report` reports usage against the plan (SolanaTracker Advanced ≈ $58–59/month USD, 200,000 requests/month — an accounting/ops metadata note, NOT a PnL/EV/profit feature). - The Kalshi↔Polymarket cross-venue observation (POLY-002) is **read-only semantic matching + MEASUREMENT only**: a deterministic normalizer + matcher over already-persisted Kalshi markets/snapshots and POLY-001 polymarket markets produces `match_label` verdicts (`comparable_market_candidate` / `unresolved_semantic_match` / `incompatible_outcome` / `incompatible_resolution` / `low_confidence_match`) for human review and measures observable differences — midpoints on a 0–1 probability scale, spreads, liquidity proxies, and `observed_difference` = |kalshi_midpoint − polymarket_midpoint| (a measured probability-point gap). - The follow-through diagnostic (FOLLOWTHROUGH-001) is **read-only measurement ANALYSIS only**: it explains why edge-precheck gap follow-through is negative by measuring the market's own movement around our measurement moments (pre-measurement move vs gap direction, post-measurement continuation/reversal, signal/forecast/snapshot ages, spread/liquidity drift) and labelling cohorts with deterministic failure-mechanism verdicts (`adverse_selection_candidate`, `stale_or_chasing_move`, `measurement_artifact_possible`, `too_thin`, `promising_needs_more_sample`, `neutral`). A verdict is a statement about MEASUREMENT QUALITY — never advice, never a signal, never an instruction; closure numbers are market movement, not PnL. It changes no edge-precheck gate, no forecast, no promotion, no MarketOps/EDGE-AUTO behavior; derives everything on demand from persisted rows; persists nothing; makes no external call; adds no flag/timer/migration. No EV, paper trading, trade recommendations, sizing, orders, wallets/private keys, signing, swaps, execution, or autonomy.
- The tick-aggregation hardening (OPS-013) is **operational storage/durability work only**: per-sub-window commits (the SQLite write lock is held for seconds, never one long transaction), bounded apply-and-commit retries on a locked database (the unit re-applies its upserts on every attempt — a rollback discards pending work, so retrying only the commit would silently lose data; this is tested), loudly-recorded failed/oversized windows (never silent; idempotent reruns repair them), and a `tick_aggregation_runs` audit spine. **The scheduled path is double-gated**: the systemd user timer artifacts are NOT auto-installed, and `aggregate-market-ticks --scheduled` no-ops unless `ENABLE_TICK_AGGREGATION_TIMER=true` (default false; manual runs always allowed). **Raw tick retention is UNCHANGED and no raw tick is deleted** — the readiness section of `tick-aggregation-report` (coverage>=0.98 over 72h, >=5 clean scheduled cycles, no recent errors, raw feed fresh) is EVIDENCE for a future, separately accepted reduction milestone and enacts nothing. Aggregated buckets remain telemetry summaries, never trading signals; no EV, paper trading, trade recommendations, sizing, orders, wallets/private keys, signing, swaps, execution, or autonomy; MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket/MEME-MAS behavior unchanged.
- The tick aggregation layer (OPS-012) is **operational storage/durability work only**: it rolls raw `market_price_ticks` — the dominant SQLite growth driver — into fixed-interval `market_price_tick_buckets` (OHLC midpoint, open/close bid/ask, spread/liquidity ranges, tick counts) so history survives at a fraction of the storage. **Aggregated buckets are telemetry summaries, never trading signals**: no side, size, EV, dollar, profit, action, recommendation, arbitrage/arb, order, wallet, signing, or execution field exists by construction, and no consumer treats a bucket as advice. Aggregation **never deletes raw ticks** (only the explicitly-invoked retention service prunes, on its own windows); **raw tick retention is UNCHANGED by OPS-012** — the tick-aggregation-report only STAGES a recommendation (a future milestone may reduce raw retention toward 24-48h once bucket coverage is proven healthy), and enacting it requires explicit human acceptance. Buckets age out on their own 90d window. Reruns are idempotent; truncation at the bounded row cap is reported, never silent; missing midpoints yield NULL OHLC, never fabricated values. It computes no EV, does no paper trading, recommends no trades, sizes nothing, places no orders, and uses no wallets/private keys/signing/swaps/execution; watcher/MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket logic is unchanged (only additive db-growth/retention/report plumbing).

- The cross-venue matcher precision hardening (POLY-PRECISION-001) is **read-only semantic and MIDPOINT-CORRECTNESS work only**. It fixes two POLY-002 defects: a Polymarket midpoint read from `outcome_prices[0]` regardless of which proposition that outcome names (the market book prices `outcomes[0]`, which is a TEAM NAME in ~26% of markets, while Kalshi encodes a game market's Yes side in the TICKER SUFFIX, not the title), and "O/U 2.5" classifying as `yes_no` because punctuation was stripped before over/under detection. A Polymarket midpoint — and therefore any `observed_difference` — is now produced ONLY when the outcome side has been explicitly aligned to the Kalshi YES proposition; otherwise both are ABSENT and the pair is annotated `outcome_side_uncertain` / `midpoint_side_uncertain` and left `unresolved_semantic_match`. It adds deterministic compatibility gates (outcome type, market scope — player prop vs game vs tournament future, over/under and handicap thresholds, entity-anchored scorelines, named-entity overlap, sport identity) whose rejections are recorded as `market_type_mismatch` / `threshold_mismatch` / `entity_mismatch` / `sport_or_game_mismatch`, and a `large_observed_difference_requires_review` REVIEW flag (measured gap > 0.35 with semantic confidence < 0.85). That flag is a suspicion that the MATCH is wrong — it is **never an opportunity, an edge, an arbitrage, or an action**, and a large gap alone never rejects a pair. Ambiguity always degrades to `unresolved_semantic_match`, never to `comparable_market_candidate`. It **does NOT identify arbitrage, compute EV, recommend trades, paper trade, size positions, place/cancel orders, or use wallets/private keys/signing/swaps/execution** — no side, size, EV, dollar, profit, action, recommendation, arbitrage/arb, order, or wallet field exists by construction. It makes no external call (matches persisted rows), installs no timer, adds no flag and no migration, and changes no MarketOps/EDGE-AUTO/MEME-NEWS/SolanaTracker/Polymarket-scan behavior.

It **does NOT identify arbitrage, compute EV, recommend trades, paper trade, size positions, place/cancel orders, or use wallets/private keys/signing/swaps/execution** — no side, size, EV, dollar, profit, action, recommendation, arbitrage/arb, order, or wallet field exists by construction.
